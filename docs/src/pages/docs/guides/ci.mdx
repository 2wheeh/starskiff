# CI Setup [Run starskiff tests on GitHub Actions]

**Prefer images.** starskiff is [image-first](/docs/guides/docker): an instance backed by a usable chain image needs no CI setup at all. Docker is preinstalled on GitHub runners, the image is pulled on demand, and nothing is provisioned.

```yaml
- run: pnpm install --frozen-lockfile
- run: pnpm vitest run # image-backed instances just work; images pulled on demand
```

`Instance` runs an image starskiff publishes — but until that image's [first publish](/docs/guides/docker#images-we-publish), it's provisioned as a binary too (the harness pins `binary: 'evmd'`); once published it joins the no-setup group.

The rest of this guide covers the **exceptions** — instances with no usable upstream image, which stay binary-provisioned (chain nodes, plus the `hermes` relayer):

| Instance | Why it's a binary exception                             |
| -------- | ------------------------------------------------------- |
| `gaiad`  | official image lags mainnet (v25/v26 vs live v27.5.0)   |
| `hermes` | an IBC relayer, run as a host binary next to the chains |
| `evmd`   | _transitional_ — until its published image exists       |

Getting their binary onto the runner is the only CI-specific work. (A local `binary` or `image` override is the [escape hatch](/docs/guides/docker#escape-hatches) for development; starskiff's own CI uses the image defaults, provisioning a binary only for the exceptions above.)

## Binary-provisioned exceptions

For the chains above, prefer prebuilt binaries over building from source: a chain binary is a large Go program, and compiling one per CI run spends the startup advantage that makes starskiff worth using. Build (or download) each binary **once**, store it where CI can fetch it, and every run afterwards pays only a seconds-long download.

### Prefer prebuilt over source-build

Store binaries as GitHub release assets and download them at the start of the job. This is what starskiff's own CI does — versions are pinned in a config file, a manual [`release-binaries.yml`](https://github.com/2wheeh/starskiff/blob/main/.github/workflows/release-binaries.yml) workflow builds and uploads them once, and [`ci.yml`](https://github.com/2wheeh/starskiff/blob/main/.github/workflows/ci.yml) downloads them per run:

```yaml
name: Test

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v6

      - uses: pnpm/action-setup@v5
      - uses: actions/setup-node@v6
        with:
          node-version: 24
          cache: pnpm

      - name: Download chain binaries
        env:
          GH_TOKEN: ${{ github.token }}
        run: |
          gh release download v27.5.0 --repo cosmos/gaia \
            --pattern "gaiad-v27.5.0-linux-amd64" --dir /tmp
          install -m755 /tmp/gaiad-v27.5.0-linux-amd64 /usr/local/bin/gaiad
          gaiad version

      - run: pnpm install --frozen-lockfile
      - run: pnpm vitest run
```

Some upstreams publish official release binaries you can download straight instead of building anything (e.g. the gaia chain, the hermes relayer):

```yaml
- name: Download hermes
  env:
    GH_TOKEN: ${{ github.token }}
  run: |
    gh release download v1.13.3 --repo informalsystems/hermes \
      --pattern "hermes-v1.13.3-x86_64-unknown-linux-gnu.tar.gz" --dir /tmp
    tar -xzf /tmp/hermes-*.tar.gz -C /tmp
    mv /tmp/hermes /usr/local/bin/hermes
```

:::note
Chain binaries are platform-specific — assets built for `linux-amd64` won't run on macOS runners or `arm64`. Match the asset to your runner, and rebuild assets when you bump pinned chain versions.
:::

`actions/cache` keyed on the binary version works too if you'd rather not manage a release.

### Build from source per run

Simplest to set up and always reproducible, but each cold run spends minutes in the Go compiler (the Go module/build cache softens repeat runs):

```yaml
- uses: actions/setup-go@v6
  with:
    go-version: '1.25'
    cache: true

- name: Build evmd
  run: |
    git clone --depth 1 https://github.com/cosmos/evm.git /tmp/cosmos-evm
    cd /tmp/cosmos-evm/evmd && go build -o /usr/local/bin/evmd ./cmd/evmd/
```

Reasonable for a repo that runs CI rarely, or as the bootstrap step that produces the prebuilt assets above.

## Split unit and integration jobs

Keep binary-free tests runnable everywhere by splitting vitest [projects](https://vitest.dev/guide/projects) — a `unit` project that never touches a chain, and an `integration` project that boots instances. CI then runs `unit` on every push and gates `integration` behind the binary-download job:

```yaml
jobs:
  ci:
    steps:
      # ... setup, no binaries needed
      - run: pnpm vitest run --project unit

  integration:
    needs: ci
    steps:
      # ... setup + binary download
      - run: pnpm vitest run --project integration
```

## Timeouts

CI runners are slower than laptops — first block can take noticeably longer than the local 3–5s. Bump the instance timeout in CI:

```ts
const instance = Instance.simd({ chainId: 'test-1' }, { timeout: process.env.CI ? 120_000 : 60_000 });
```

The same applies to vitest's `hookTimeout` if you boot instances in `beforeAll`/`globalSetup`.
